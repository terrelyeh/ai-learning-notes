# SecureMark 證件照浮水印 — 開發紀錄

> 🤖 **我用 AI 做了什麼**：從零建立一個純前端的證件照浮水印工具，包含 PDF 多頁處理、拖曳調整、列印尺寸等完整功能
> ⏱ **沒有 AI 的話**：自己從 Canvas API 文件開始研究，加上 PDF 解析和匯出，估計需要 2–3 天
> ✅ **最終成果**：已部署上線的 Web App，支援圖片與多頁 PDF，所有處理在瀏覽器端完成，零伺服器成本

> 台灣租屋、開戶、求職都要交身分證影本，加浮水印是常見的自保方式。市面上的工具不是要上傳照片就是功能陽春，所以決定自己做一個純前端版本。

---

## 一、為什麼要做這件事

核心需求很簡單：**在證件照上加半透明浮水印文字，標明用途，防止被挪用**。

但「簡單需求」不代表「簡單實作」。真正要解決的問題是：

- **隱私**：證件照不能上傳到任何伺服器，必須純前端處理
- **實用性**：要支援 PDF（很多人掃描證件存成 PDF），而且要能多頁處理
- **列印**：加完浮水印後要能輸出成 A4 PDF，甚至指定證件尺寸列印
- **易用**：非技術用戶也要能快速完成，不能太多設定項

這些需求疊起來，就不是一個「用 Canvas 畫文字」能解決的事了。

## 二、技術選型

| 工具 | 選的理由 |
| --- | --- |
| React 19 + TypeScript | 元件化控制面板，型別安全避免設定值出錯 |
| Vite 7 | 開發快、建置快，HMR 體驗好 |
| Tailwind CSS | 快速迭代 UI，不用寫獨立 CSS 檔案 |
| Canvas API | 瀏覽器原生，不需額外 library 就能繪製浮水印 |
| pdfjs-dist | Mozilla 維護的 PDF 解析器，把 PDF 頁面渲染成 Canvas |
| jsPDF | 把加完浮水印的 Canvas 重新打包成 PDF 下載 |
| Gemini API（可選） | AI 根據用途自動建議浮水印文字，後來判斷非必要，降為可選功能 |

> **選型原則**：能在瀏覽器跑的就不要靠伺服器。PDF 的「讀」和「寫」是兩件事，需要兩個不同的 library（pdfjs-dist 負責讀，jsPDF 負責寫），這一點在選型時容易忽略。

## 三、AI 協作策略

這個專案的分工模式很明確：

**AI 負責**：
- 產生完整的專案骨架（React + Vite + TypeScript 配置）
- 實作 Canvas 浮水印繪製邏輯（平鋪計算、旋轉矩陣）
- PDF 解析與重新打包的完整流程
- UI 元件與控制面板的 Tailwind 樣式

**人負責**：
- 定義需求優先順序（先做核心功能，再加進階功能）
- 測試每一版成果，發現 UI 問題和 edge case
- 判斷功能取捨（例如：移除 AI 寫作功能，因為增加複雜度但價值不大）
- 驅動 UI 重構方向（從卡片式 → badge 式 → 最終的精簡版）

**提問模式**：主要是「範例驅動型」— 先讓 AI 產出一版，測試後描述具體問題，再逐步修正。不是一開始就寫完整規格，而是透過迭代逼近想要的結果。

## 四、推進方式

從 git history 可以看出清楚的三個階段：

### 階段一：骨架與核心功能（4 commits）

讓 AI 產出完整的專案結構和基礎功能。第一版就包含了：上傳圖片、Canvas 繪製浮水印、基本設定面板、下載 PNG。這個階段 AI 的產出幾乎可以直接用，人只需要微調預設值（例如把透明度從太高調低）。

### 階段二：功能擴充（1 個大 commit）

一次性加入 5 個功能：模板快選、自動日期、前後對比預覽、字體選擇、多頁 PDF 支援。這是整個專案最密集的一次推進。

技術上最有趣的是多頁 PDF 的處理方式 — 用 pdfjs-dist 把每一頁渲染成獨立的 `HTMLImageElement`，存在 state 裡，預覽時只顯示第一頁，下載時遍歷所有頁面：

```typescript
// 把 PDF 每一頁都渲染成圖片存起來
const pages: HTMLImageElement[] = [];
for (let pageNum = 1; pageNum <= pdf.numPages; pageNum++) {
  const page = await pdf.getPage(pageNum);
  const viewport = page.getViewport({ scale: 3.0 }); // 高解析度
  // ... 渲染到 tempCanvas，轉成 Image
  pages.push(img);
}
setPdfPages(pages);
setImage(pages[0]); // 預覽第一頁
```

### 階段三：UI 打磨與 Bug 修復（8 commits）

這是花最多來回的階段。控制面板經歷了三次重構：

1. **初版**：功能全塞在一個長面板裡 → 太雜亂
2. **卡片式**：每個功能區塊獨立成卡片 → 視覺更清楚，但佔空間
3. **精簡版**：Section 分區 + 藍色 accent header → 最終版本

這個階段的教訓是：**UI 的迭代成本比功能開發高**。AI 可以很快產出功能邏輯，但「好不好用、好不好看」需要人反覆測試和調整。

## 五、踩到的坑

### Bug 1：切換排版模式後浮水印消失

- **症狀**：從「滿版鋪滿」切到「單一置中」模式，畫面上看不到浮水印
- **根本原因**：`offsetX` / `offsetY` 在滿版模式下可能被拖曳到很大的值，切到單一模式後，浮水印被定位到畫面外
- **解法**：切換模式時重置 offset

```typescript
// 排版按鈕的 onClick
onClick={() => updateSettings({ style: mode, offsetX: 0, offsetY: 0 })}
```

> **帶走的原則**：不同模式共用 state 時，要考慮模式切換時的 state 重置。這種 bug AI 不會主動發現，因為它不會「操作」UI。

### Bug 2：手機上控制面板被截斷

- **症狀**：手機瀏覽時，控制面板底部被切掉，無法滾動到下載按鈕
- **根本原因**：`h-full` 和 `overflow-y-auto` 在手機上讓面板高度被限制在視窗內，但內容溢出
- **解法**：這兩個屬性只在 `lg` 斷點以上才套用

```tsx
// 修正前
<div className="flex flex-col gap-3 ... h-full overflow-y-auto p-3">

// 修正後
<div className="flex flex-col gap-3 ... lg:h-full lg:overflow-y-auto p-3">
```

> **帶走的原則**：桌面版的 layout 策略（固定高度 + 內部滾動）在手機上幾乎一定會出問題。響應式設計要從手機端開始思考。

### Bug 3：AI 寫作功能的取捨

- **症狀**：不算 bug，而是功能判斷 — 原本加了一個「AI 幫你寫浮水印文字」的功能（用 Gemini API），但實際使用後發現：模板快選已經涵蓋 90% 的場景，AI 建議反而增加等待時間和 API key 設定的門檻
- **決策**：移除 AI 寫作的主動觸發，降級為可選的 service，保留程式碼但不在 UI 曝光

> **帶走的原則**：「能做」不等於「該做」。加功能之前先問：這比現有方案（模板快選）好多少？如果改善幅度不大但增加複雜度，就不要加。

## 六、最終樣貌

一個單頁 Web App，左邊是 Canvas 預覽區（支援拖曳調整位置），右邊是控制面板（文字、樣式、輸出三個區塊）。功能包含：

- 上傳圖片或多頁 PDF
- 6 種預設模板快選（租屋、開戶、求職等）
- 自訂文字、字體、顏色、透明度、角度、間距
- 滿版鋪滿 / 單一置中兩種排版
- 自動加入今日日期
- 前後對比預覽
- 下載 PNG 或 PDF（A4 / 證件尺寸 / 護照尺寸）

已部署在 Vercel，push main 自動部署，零伺服器成本。

## 七、如果繼續往下

- **多語系**：目前只有繁體中文，但浮水印工具的概念是通用的，加英文 / 日文介面就能擴大受眾

## Takeaway

### 這個案例展示了什麼

「讓 AI 寫第一版，人來迭代」的協作模式在工具類專案特別有效。AI 擅長的是把明確的技術需求（Canvas 繪製、PDF 解析）快速實作出來；人擅長的是在實際操作中發現 edge case（模式切換 bug、手機截斷）和做功能取捨（移除 AI 寫作）。

14 個 commits 裡，前 6 個是 AI 主導的功能建設，後 8 個是人驅動的品質打磨。**功能可以一次到位，但 UI 和體驗需要反覆磨**— 這個比例（40% 建設 / 60% 打磨）在 AI 協作開發中很典型。

### 可移植性

這個方法適用於任何「需求明確、技術選型成熟」的前端工具。如果你的專案涉及大量創意決策（品牌設計、文案風格）或複雜的後端邏輯（權限、交易），AI 的第一版產出可能需要更多人工修改。

### 如果你也要做類似的事，先問 AI 什麼

1. **「我要做一個 [功能描述] 的純前端工具，用 React + TypeScript + Vite，幫我產出完整的專案結構和核心功能」** — 讓 AI 一次產出可運作的骨架，比分步驟問效率高很多
2. **「這個功能在手機上會有什麼問題？」** — 每次加完功能後主動問，比自己測試後才發現要快
