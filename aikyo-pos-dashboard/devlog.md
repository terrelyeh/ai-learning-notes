# 愛嬌青春 POS 儀表板 — AI 協作開發紀錄

> 🤖 **我用 AI 做了什麼**：從零打造一個完整的 POS 營運分析儀表板，包含圖表視覺化、Admin 管理介面、一鍵發布到雲端
> ⏱ **沒有 AI 的話**：需要找工程師外包或學會前端框架，至少數週到數月，費用數萬元起
> ✅ **最終成果**：一個單檔 HTML 儀表板，部署在 Cloudflare Pages，每月只需上傳 CSV 就能更新，30 秒自動上線

> 愛嬌青春製冰室需要一個方式來理解每月營運狀況，但 POS 系統（電小二 Vital+）只能匯出原始 CSV。這篇紀錄了如何用 AI 在一週內，從零建出一個可持續使用的營運儀表板。

---

## 一、為什麼要做這件事

開店後，每個月都會從 POS 系統匯出營運資料 — 營收多少、哪些品項賣得好、客人用什麼方式付款。但這些資料散落在四份不同的 CSV 檔案裡，打開來就是一堆數字，完全看不出趨勢。

核心痛點：

- **資料分散**：營運報表、支付明細、品項統計、銷售日報表，四份 CSV 各自獨立
- **無法視覺化**：POS 後台只有基本表格，沒有圖表，看不出跨月趨勢
- **分享困難**：想跟合夥人分享當月表現，只能截圖或傳 Excel
- **成本考量**：市面上的 BI 工具（如 Tableau、Power BI）對小店來說太重、太貴

我需要的其實很簡單：**一個網頁，上傳 CSV 就能看圖表，能分享連結給合夥人**。

## 二、技術選型

| 選擇 | 決定 | 理由 |
|------|------|------|
| 架構 | 單一 HTML 檔案，資料內嵌 | 零基礎設施成本，不需後端也不需資料庫 |
| 圖表 | Chart.js 4.4 | 免費、輕量、社群成熟 |
| 部署 | Cloudflare Pages | 免費靜態託管，GitHub push 後自動上線 |
| 發布方式 | 內建 GitHub API push | 不需要開終端機，在網頁上按一個按鈕就能發布 |
| 資料前處理 | Claude Code skill | 銷售日報表需從原始訂單 Excel 轉換為每日品項數量 CSV |

> **選型原則**：盡可能不依賴任何需要維護的服務。一個 HTML 檔 + 一個免費 CDN，就是全部。

## 三、AI 怎麼幫我做到的

### 分工方式

整個開發過程中，我負責的是「告訴 AI 我要什麼」和「看結果、給回饋」，AI 負責把想法變成可以跑的程式碼。

**AI 負責：**
- 所有 HTML / CSS / JavaScript 程式碼撰寫
- Chart.js 圖表配置與資料綁定
- CSV 解析邏輯（不同報表格式的欄位對應）
- GitHub API 串接（實現網頁內一鍵發布）
- 響應式設計（手機、平板、桌面適配）

**我負責：**
- 決定要看哪些指標（營業額、來客數、客單價、品項份數）
- UI 風格偏好（方眼紙背景、品牌粉色、卡片式佈局）
- 業務邏輯判斷（哪些品項歸哪個分類、付款方式怎麼分）
- 資料正確性驗證（對照 POS 原始數字）
- 功能優先級（先做基本圖表，再加分享、通知模板等）

### 推進方式

我的做法是**逐步堆疊，每次只加一件事**：

1. 先做出最基本的月報頁面 — KPI 卡片 + 一張折線圖
2. 確認數字正確後，加上品類分佈圓餅圖和品項排行
3. 加入月份切換（Tab 式導航）
4. 加入趨勢總覽（跨月比較）
5. 加入 Admin 模式（上傳 CSV、發布功能）
6. 最後做分享、通知模板、深度連結等「好用」功能

每一步都是：**我描述需求 → AI 寫出來 → 我看結果 → 微調**。43 次 commit，大約就是 43 輪這樣的迭代。

### 關鍵轉折

**轉折一：從「下載 HTML」到「一鍵發布」**

一開始，每次更新月份資料後，需要下載整個 HTML 檔案，再手動用 git push 推上去。這個流程對非工程師來說太麻煩了。後來我請 AI 直接在網頁裡串 GitHub API，實現按一個按鈕就把更新後的 index.html push 到 GitHub，Cloudflare 自動部署。這讓整個「每月更新」流程從需要開終端機變成只需要開瀏覽器。

**轉折二：銷售日報表需要格式轉換**

四種報表中，前三種（營運報表、支付明細、品項統計）從 POS 匯出就是 CSV，可以直接上傳。但第四種「銷售日報表」匯出的是原始訂單 Excel — 每一列是一筆訂單，品項全部擠在同一欄裡。儀表板需要的是「每天每個品項賣了幾份」的格式，兩者差很多。

我用 Claude Code 建了一個 `sales-report-parser` skill，自動把原始訂單 Excel 轉成每日品項數量的樞紐表 CSV。只要提供檔案路徑，它就會解析訂單、拆解品項、按日期彙總，產出儀表板能直接吃的格式。

**轉折三：UI 調校的密集迭代**

git 歷史裡有大量 UI 相關的 commit（字體大小、間距、背景樣式），這些都是在瀏覽器裡看了覺得「這裡不太對」，然後描述給 AI 調整的。這種視覺微調特別適合 AI 協作 — 你不需要知道 CSS 語法，只需要說「這個字太小了」「間距再大一點」。

## 四、最終成果

一個部署在 `aikyo-pos.pages.dev` 的營運分析儀表板，功能包含：

- **月度總覽**：營業額、來客數、客單價、營業天數，附與上月比較
- **每日趨勢圖**：折線圖 + 長條圖，可切換營業額 / 來客數 / 客單價 / 出杯數
- **分佈統計**：品類和付款方式的環形圖
- **品項排名**：橫條圖 + 可排序明細表
- **趨勢總覽**：跨月比較頁面，含洞察分析
- **Admin 模式**：上傳 CSV、一鍵發布、通知模板（LINE 格式）、分享連結
- **深度連結**：可透過 URL 參數直接開啟特定月份

整個應用是一個 2,600 行的 `index.html`，沒有後端、沒有資料庫、沒有框架。每月維護成本：$0。

## 五、踩到的坑

### 銷售日報表的格式落差
前三種報表（營運報表、支付明細、品項統計）從 POS 匯出就是乾淨的 CSV，直接上傳沒問題。但銷售日報表匯出的是原始訂單 Excel — 每列一筆訂單，品項資訊全部塞在同一欄、用換行分隔，格式像 `2 x 冰品-芒果冰沙 (甜度: 正常)`。儀表板需要的是「日期 × 品項 × 數量」的樞紐表，兩者差了一整個轉換步驟。最後建立 `sales-report-parser` skill 自動完成拆解與彙總。

> **帶走的原則**：POS 匯出的格式不一定是你要的格式。先確認「系統給你的」和「你實際需要的」之間有沒有落差，有的話就建一個自動化的轉換流程。

### 單一 HTML 檔案的取捨
把所有東西（CSS、JS、資料）塞進一個 HTML 檔的好處是部署極簡，但檔案會越來越大。目前三個月的資料，檔案約 120 KB，還可以接受。但如果累積到 12 個月以上，可能需要考慮拆分資料。

> **帶走的原則**：小店小量資料，單檔夠用。但要有心理準備，資料量大了之後需要重構。

### 發布按鈕的權限控制
一開始「發布」按鈕在一般模式也看得到，任何人都能按。後來改成只在 `?admin` 模式（需要密碼）才顯示，避免訪客誤觸。

> **帶走的原則**：任何會改動線上內容的操作，一定要做權限控制，即使是很小的專案。

## 六、如果繼續往下

- 資料量成長後，可以把月份資料拆成獨立 JSON 檔案，index.html 按需載入，避免單檔膨脹
- 加入損益平衡分析 — footer 已經預留了連結，把成本資料加進來就能算出真正的盈虧
- 做成 Progressive Web App（PWA），讓手機可以像 App 一樣開啟
- 把 `sales-report-parser` skill 做成網頁版，讓不用 Claude Code 的人也能轉換銷售日報表

## Takeaway

### 這個案例展示了什麼

不需要會寫程式，也能用 AI 從零建出一個「真的可以用」的營運工具。關鍵不在技術能力，而在於：

1. **你清楚知道自己要解決什麼問題**（看不懂 POS 匯出的數字）
2. **你能描述你想要的樣子**（要有圖表、要能分享、要能自己更新）
3. **你願意一輪一輪地迭代**（43 次 commit 代表 43 次「看了再改」）

### 可移植性

這個方法適用於任何「有固定格式的資料來源 + 需要視覺化呈現」的情境，例如：
- 餐廳的月營運報表
- 電商的銷售數據儀表板
- 團隊的 KPI 追蹤頁面

不適用於：需要即時資料更新、多人同時編輯、或資料量非常大的場景。

### 如果你也要做，先問 AI 什麼

1. 「我有一份這樣格式的 CSV（貼範例），幫我做一個網頁儀表板，顯示 [你想看的指標]」
2. 「這個儀表板要能部署到免費的靜態網站託管服務，不需要後端」

從這兩個 prompt 開始，就能啟動整個流程。
