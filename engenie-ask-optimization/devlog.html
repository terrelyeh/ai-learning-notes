問題解決 · RAG／LLM 效能

把 RAG 問答從 6 秒壓到 3 秒

跟 Claude 一起 code review 一個內部 RAG 問答系統（EnGenie Ask），診斷出延遲的真正來源、系統性地修，並把整套經驗萃取成一份下次能直接對照的 RAG／LLM 實作清單。

整理日期：2026-07-05  ·  類型：問題解決 / AI 協作
我用 AI 做了什麼 跟 Claude 一起 code review 內部 RAG 問答系統，把回覆延遲從 5–6 秒壓到約 3 秒，並補上一整批資安 hardening。
沒有 AI 的話 得自己逐段 profiling、翻三家 LLM 的 streaming／thinking 參數文件、逐一比對 migration 與 prod 的 RLS 現況——大概要好幾天。
最終成果 三批改動全上 prod（來源約 2 秒出現、首字約 3.5 秒），並萃取出一份可重用的 RAG／LLM 實作清單。
01

為什麼要做這件事

EnGenie Ask 是公司內部的 RAG 問答模組——基於自有知識庫的 AI 問答。用起來一直有個痛點：每次問問題，都要等五、六秒才開始有回覆。對內部工具來說，這個等待感很傷體驗，使用者會懷疑「是不是壞了」。

我想知道三件事：功能上有沒有值得優化的地方、架構或資安有沒有疑慮，以及最關鍵的——這個延遲有沒有辦法縮短。

!
這不是寫新功能，而是一次「請 AI 幫我體檢一個已上線系統」。重點是診斷、系統性地修、並把經驗固化成可重用的清單。
02

慢在哪：一條「串行鏈」

AI 幫我把 /api/ask 的處理流程逐步攤開、逐段量時間後，真相很清楚：5–6 秒不是「某一步很慢」，而是七、八個步驟全部排隊在串行等。

驗證身分
  └─▶ 讀 OpenAI 金鑰 (打 DB)
      └─▶ 算問題向量 (打 OpenAI，東京→美國)
          └─▶ 向量搜尋
             └─▶ 型號補充查詢 (逐一掃全表)
                 └─▶ 讀 persona (打 DB)
                    └─▶ 讀 LLM 金鑰 (又打 DB)
                       └─▶ 【 Gemini 先思考 7–15 秒 】
                           └─▶ 才開始吐字

其中最大的一塊還是白等：Gemini Flash 這類模型在吐出第一個字之前，會先在伺服器端「思考」，而這段時間使用者只看到空白轉圈。對「把查到的資料整理成答案」這種任務，深度推理根本沒有加值。

→
關鍵認知：延遲是一條鏈，不是一個點。要先量每一步、再優化，不能憑感覺猜瓶頸——真正最大的那塊（思考階段），憑讀 code 完全看不出來。
03

AI 怎麼幫我做的

分工方式：我出方向與決策，Claude 負責掃程式碼、逐段量延遲、實作、寫 migration、跑驗證。過程中它還會 spawn 平行的 subagent 同時做資安掃描和架構掃描，等於一次多路體檢。

AI 負責
  • 攤開處理流程、逐段實測每一步耗時
  • 平行 subagent：同時跑資安掃描 + 架構掃描
  • A／B／C 對照驗證假設（關思考會不會傷品質）
  • 實作改動、寫 migration、跑 build 與端到端驗證
人負責
  • 出方向：一句「太慢，有辦法縮短嗎」
  • 風險決策：要不要動 prod、哪些可接受
  • 定品質底線：答案不能因為求快而變差
  • 拍板每批改動是否合進 main（自動部署）
提問模式：症狀描述型

我沒有先假設答案，而是直接描述現象——「回覆要五六秒，有辦法縮短嗎」——讓 AI 去推斷原因、拆解瓶頸。這種問法的好處是不會把 AI 的注意力鎖死在我先入為主的方向上。

三個關鍵轉折
◆轉折一：從「猜」到「量」

AI 沒有直接改 code，而是先用真實的 prod 資料組出跟線上一樣的 prompt，實測每一步耗時。這一量才發現 Gemini 的思考階段吃掉 7–15 秒——這是憑讀 code 看不出來的隱形瓶頸。

◆轉折二：A／B／C 實測驗證假設

我對「關掉思考會不會讓答案變差」有疑慮。AI 就直接跑三組對照（預設／低思考／關閉），逐字比對答案品質後才確認可以安全關掉：首字 18 秒 → 1.2 秒，品質不變。

◆轉折三：一句話帶出更大的隱憂

修完效能後，AI 主動指出對話紀錄沒有綁使用者——任何同事都能讀別人的對話。這不是我原本問的，但比效能更嚴重，順勢就一起修掉了。

04

做了哪些優化

改動分三批，全部開在 feature branch、驗證後才合進 main（main 會自動部署 prod）。

優化項在做什麼效果
關掉 Gemini Flash 思考Flash／Lite 帶 thinkingBudget:0（Pro 保留）首字 18s → 1.2s，單一最大改善
檢索平行化persona／金鑰／輔助提示跟向量搜尋同時跑省 0.5–1s
三層快取金鑰、persona、問題向量（TTL／LRU）重複問題首字 4.1s → 0.6s
型號查詢加索引pg_trgm GIN，ilike '%型號%' 不再掃全表省 0.1–0.5s，隨資料成長更明顯
來源卡片提前送檢索一完成就推來源，不等 LLM 講完體感：2 秒就看到「找到什麼」
history 字元預算對話歷史設上限，不再無限脹 prefill聊久了也不變慢
🔒
同批補上的資安 hardening：對話綁使用者（原本任何人可讀刪他人對話）、passcode 端點防暴力猜測、RLS 現況補進 migration、知識抓取的 SSRF 防護，另加 markdown Error Boundary（畸形回答不再白屏整個聊天）。

06

萃取：RAG 實作該注意的事

不管 RAG 與否，串 LLM 的請求大致都是這個形狀——把它記住，優化就有地方下手：

Guard
Auth / 限流
放最前面，擋掉不該進來的
Prepare
平行準備
金鑰/設定/(RAG:檢索) 全部 Promise.all + 快取
Stream
串流 LLM
逐字 SSE + 中間訊號（來源提早送）
After
後處理
存紀錄/算 metadata，別擋住回應
通用（跟 RAG 無關）
  • 延遲是串行鏈——先量每一步，別猜瓶頸。
  • 推理預算匹配任務——查答案／摘要類關掉 thinking，需要多步推理才開。
  • 平行化不相依的準備工作；快取金鑰／設定／查詢向量（記得在寫入端 invalidate）。
  • 串流 + 中間訊號：狀態事件 +「來源提早送」是 CP 值最高的體感優化。
  • 控制 context／history 大小——prefill 成本每輪都在漲。
RAG 專屬
  • 向量索引選對——HNSW 通常比亂設 lists 的 ivfflat 召回好、免調參。
  • 字面查詢也要索引——型號 ilike '%…%' 沒索引 = 全表掃描，用 pg_trgm。
  • embedding 只用當前問題——串歷史會讓前一題主題污染這題的向量搜尋。
  • scope 過濾密不透風——有補充查詢繞過主查詢時，最後要再套一次過濾。
資安 / 正確性
  • per-user 資料綁 user_id；公開端點限流；ingest 擋 SSRF。
  • 金鑰放 header 不放 URL；錯誤訊息 redact 後再 log；schema 改動補 migration。
07

換成「非 RAG」的純 LLM 應用，一樣嗎？

大部分原則一樣，因為它們是關於「怎麼呼叫 LLM、怎麼編排請求」，跟 RAG 無關：平行化、快取、串流、thinking budget、控制 context、限流、秘密處理，還有上面那個四段骨架都照舊。RAG 專屬的（向量索引、embedding 快取、檢索污染）會拿掉，但多出幾個新槓桿：

新槓桿為什麼重要
Prompt Caching固定的長 system prompt／context 可快取前綴，大幅降 TTFT 與成本。RAG 每次 context 都不同、較難吃到；固定 prompt 的應用吃得最滿，該列為優先。
Tool calling 往返成本每輪工具呼叫都是一次完整 LLM 往返，延遲會疊加。優化重點變成減少來回次數（批次、平行工具、一次給足資訊）。
結構化輸出JSON schema／structured output 有自己的延遲與可靠度考量，跟自由文字不同。
→
一句話：「怎麼跟 LLM 說話」的優化通用；「怎麼找資料餵給它」才是 RAG 專屬。換成一般 LLM 應用，把檢索那塊換成「你的 context 從哪來」，其餘照舊，再把 prompt caching 加進優先清單就對了。
!
prompt caching、reasoning 參數的具體設定與計價各家不同也常改版，實作時直接查該 provider 當下的 API 文件，別靠記憶。
08

Takeaway

這個案例展示的思路

用「症狀描述型」提問，把一個模糊的抱怨（「太慢」）丟給 AI 去拆解，而不是自己先假設答案。最有效的一步是讓 AI 先量、再改——真正的瓶頸（Gemini 思考）憑讀 code 完全看不出來，是量出來的。

可移植性

這套「請 AI 體檢已上線系統」的方法，適用於任何有明確痛點但原因不明的效能／資安問題。「先用真實資料量每一步」和「改動前先 A／B 驗證假設」這兩個習慣跟技術棧無關，任何專案都能複製。前提是能提供真實環境的存取（DB、prod URL），AI 才量得準。

如果你也要做，先問 AI
→「幫我把這個 API 的處理流程逐步攤開，量出每一步的耗時，找出瓶頸在哪。」
→「這個瓶頸如果要改，先幫我跑一組 A／B 對照，確認改了不影響輸出品質再動手。」
