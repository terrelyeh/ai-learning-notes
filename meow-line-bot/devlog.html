喵哥 LINE 機器人 × 酒款管理

我是台北一間下午酒場的老闆、不是工程師。這篇不寫「怎麼跟 AI 聊天」，而是把這兩個模組——LINE 問答機器人「喵哥」＋拍照 AI 建檔的酒款資料庫——實際怎麼運作、又怎麼打通講清楚。code 是 AI 寫的，架構是我搞懂的。

整理日期：2026-07-18  ·  類型：AI 工具開發型
做了什麼 兩個打通的模組——(1) 把原本躲在 ChatGPT GPTs 裡的店務助理「喵哥」搬進 LINE 當員工問答機器人；(2) 一套「拍酒標 → AI 建檔」的選酒筆記資料庫，而它正是喵哥回答酒款問題的知識來源。
沒有 AI 的話 我不會寫程式，這兩個都等於「做不出來」。這次跟 AI 對話幾天就上線了。
成果 LINE「下午酒場 喵哥」問答 / 看圖；後台酒款資料庫 AI 拍照建檔、延伸文案、狀態篩選。
01

為什麼做（其實是同一件事的兩面）

問題一 · 喵哥搬家

店裡本來有個 ChatGPT 的 GPTs 助理「喵哥」，員工可以問它酒水、出餐、服務 SOP。但要用它得先打開 ChatGPT、還要登入，現場忙起來沒人這樣做。而員工每天都掛在 LINE 上。→ 把喵哥搬進 LINE。

問題二 · 酒款知識沒有結構

下午酒場「什麼酒都賣」、酒款一直換，但「這支酒怎麼介紹、什麼風味、怎麼推給客人」散在每個人腦裡、講法不一，新人只能用背的。→ 建一個「選酒筆記」資料庫，把每支酒的介紹方式固定下來。

交會點 · 同一件事的兩面

這兩個問題是同一件事的兩面——酒款資料庫建好的內容，就是喵哥回答酒款問題的依據。員工在 LINE 問「水芭蕉怎麼樣」，喵哥查的就是這個資料庫。所以兩個一起做最划算，也是這篇分兩大主題的原因。

02

喵哥 LINE 機器人

整支機器人就是一條「訊息進 → 組 prompt → 問模型 → 回覆」的管線。員工在 LINE 傳訊息，LINE 平台把它 POST 到我的 webhook；系統驗證來源、查身份，把「人設＋知識庫整包＋最近對話」組成一次 API 呼叫交給大語言模型；模型回答後用免費的 reply 通道傳回，並把這輪對話寫進資料庫當下次的記憶。

拆開來看，五個技術點：

① webhook 驗簽

LINE 收到訊息後 POST 到我的網址（webhook）。任何人都能打這網址，所以每個請求都帶 X-Line-Signature——用 channel secret 對內容算 HMAC-SHA256，對不上就丟掉。

② 身份與權限

純 6 碼數字走綁定（把 LINE 帳號寫進對應員工），其他文字／圖片才進問答；進問答前過兩道閘——總開關 bot_enabled、白名單 bot_allowlist（目前只開主管群）。

③ 組 prompt（最核心）

system 段＝喵哥人設＋店內知識庫（整包純文字）；user 段＝發問者身份＋這次問題（身份放 user 不放 system，是為了不污染快取）；記憶＝撈這個員工近 6 小時對話——模型本身沒記憶，「記得剛剛」是我們自己去 DB 撈回來喂的。

④ 呼叫模型

透過 OpenRouter 送給大語言模型，關掉內部「思考」、Anthropic 系模型在知識段標記快取。一次呼叫大致長這樣：

{
  "model": "anthropic/claude-sonnet-5",
  "messages": [
    { "role": "system", "content": "〈喵哥人設〉＋〈店內知識庫整包〉" },
    { "role": "user",   "content": "〈發問者〉：〈這次的問題〉" }
  ],
  "reasoning": { "enabled": false },   // 關掉思考，別燒光回覆額度
  "plugins":   [{ "id": "web" }]       // 需要時才開上網查
}
⑤ 回覆＋存檔

用 replyToken 回覆（免費，見第五節），同時把這輪對話寫進 bot_messages——既是紀錄，也是下一輪的記憶。

→
知識庫有三個入口：制度文件、上傳檔案、酒類資料庫。第三個就是下一節的主題。
03

酒款管理：拍照 AI 建檔的選酒筆記資料庫

這是這次另一個大模組，也是喵哥「懂酒」的底層。

為什麼是「資料庫」不是「一份文件」

要能標售完 / 下架、能篩選、能累積、能一支支餵給喵哥。所以每支酒是一筆結構化資料，不是一段散文。

拍照 AI 建檔流程（最省力的建檔方式）
拍酒標照上傳
  → AI 辨識酒款（看圖）＋ 上網查證（可關）＋ 照「選酒筆記框架」填成 JSON 草稿
  → 主管過目、補店內價格與出杯 ml
  → 上架

這是典型的 human-in-the-loop：AI 做粗活（辨識、查、寫草稿），人做最後把關。

為什麼要「框架」（固定欄位）

每支酒都填同一組欄位——官方描述 / 店內版風味 / 四位人設的員工心得 / 小故事×2 / 調飲方式 / 分客群推薦 / 注意事項。框架的用意是全店講法一致，而且涵蓋員工各種問法：有人問味道、有人問故事、有人問「怎麼推給怕酒味的客人」——框架把這些面向都先準備好。

防呆（不讓 AI 亂猜）

辨識能不能對，關鍵是「圖有沒有真的送到模型」（這次踩最深的坑，見第六節）。所以加了幾道 gate：圖沒送成功就中止請重傳、AI 自己沒把握就先回頭問主管——寧可停下來，也不默默填一份錯的資料。

上網查證是開關

預設開（補官方描述 / 酒造背景），但廠商的廣告宣傳文很容易混進來，所以做成可關——想快、或想純用模型知識時關掉。

延伸文案

一支酒的筆記建好後，可以一鍵長出 IG 四格 / 酒單插卡 / 豆知識小卡，語氣可選（輕鬆 / 職人 / 潮流）。同一份選酒筆記，變成好幾種對外素材。

兩個模組怎麼打通

酒款資料庫是喵哥知識庫的三入口之一。所以——你在後台建好一支酒，喵哥在 LINE 上立刻就會回答這支酒。這就是「喵哥懂酒」的來源，也是兩個主題的交會點。

05

關鍵技術點

1 · LINE 有兩種訊息：一種免費、一種計費

回覆訊息（reply）是使用者傳來、機器人用 replyToken 回應——免費、無上限。喵哥每次回答都走這條。主動推播（push）是機器人主動送的（每天班表提醒）——計費、逐人算。免費版一個月 200 則算的是 push，大約 6~7 人每天推就爆 200。所以問答可以全店免費開，全店每日推播才要評估付費——這決定了成本結構。

2 · 知識庫是「全量上下文 + prompt caching」，不是 RAG

每次對話把店裡全部知識（制度、公告、現貨酒單）組成一段純文字整包塞進 system。省錢靠 prompt caching：不變的知識段標記起來，之後重複的只算約 1/10。要吃到快取，那段內容必須逐字節穩定（所以「發問者是誰」放 user 段、不混進被快取的 system）。好處是模型看得到全貌，跨酒款比較不會因搜尋漏抓而答錯。

3 · 用 OpenRouter 做「模型路由」，換模型是改一個字串

一把金鑰、背後接所有家模型，介面統一。想從 Claude 換 GPT / Gemini 比效果，只改一個字串。我甚至把聊天模型和寫酒款文案的模型分成兩個設定。

4 · AI 建檔：human-in-the-loop

拍酒標 → AI 辨識＋查＋寫草稿 → 人補價格、把關 → 上架。AI 做粗活、人做最後決定，既省時間又不失控。這是酒款模組的核心設計，也是「AI 出錯也不會直接上線」的保險。

06

踩到的坑（都是技術問題）

⚠坑一：對話紀錄一直是空的 → 多輪記憶從沒生效（喵哥）

根本原因：把「使用者這句」和「喵哥那句」兩列一起批次 insert，但兩列欄位組不一致，被資料庫的 API 層（PostgREST）整批拒絕（PGRST102），而程式沒檢查回傳的錯誤 → 靜默失敗。

→
解法：兩列補齊同欄位組 ＋ 寫入後檢查 error。原則：「沒報錯」不等於「成功」。
⚠坑二：喵哥回「有點忙不過來」、草稿也空白 → token 被「思考」燒光（兩者共通）

根本原因：新一代模型有「深度思考（reasoning）」模式，這些思考也算進回覆的 token 額度；某些供應商預設開，額度被燒光、正文一個字都吐不出來（finish_reason=length）。

→
解法：即答場景一律帶 reasoning:{enabled:false} 關掉。
⚠坑三：改一句人設，喵哥講話擠成一坨（喵哥）

根本原因：加的一句「短問題幾行講完」被模型過度執行。

→
解法：提示詞牽一髮動全身，改完一定要拿真實問題實測輸出再上，不能憑感覺盲改。
⚠坑四：🔴 酒標辨識全錯 → 一個尾端換行害圖被默默丟掉（酒款，這次挖最深）
!
症狀：拍酒標建檔，AI 認出的酒全是錯的——明明是日本清酒「水芭蕉」，卻回「Bacardi 蘭姆酒」「三得利知多威士忌」，還自信地捏造酒標上根本沒有的文字。換任何模型都一樣。

追查：把真圖直接送模型，Kimi / Gemini / Sonnet 14/14 全部辨識正確。所以模型視力沒問題——是 app 根本沒把圖送出去，模型沒圖只能瞎猜。

根本原因：Vercel 上 NEXT_PUBLIC_SUPABASE_URL 的值尾端夾了一個換行 \n。送圖前有一道「只准抓自家圖庫」的安全檢查，用原始字串比對網址開頭；因為 env 夾了 \n，比對字串多了個換行，乾淨的圖片網址永遠對不上 → 檢查判定「不是自家圖」→ 把圖丟掉。而別處的網址都經過會自動清掉 \n 的元件，只有這道手工字串比對中招，所以只有辨識壞、登入/資料庫都正常。只在正式站中招（我本機的設定沒有這個 \n）。

→
解法：比對前先把值 trim() 乾淨（治標）＋ 清掉 env 裡的換行（治本）。帶走的兩個原則：env 的值可能夾看不見的空白/換行，做字串比對前先 trim；還有——圖沒送到時，模型不會說「我沒看到圖」，它會自信地編一個，所以要加「圖沒送成功就中止」的防線。
⚠坑五：酒款 AI 草稿「時好時壞」噴解析錯誤 → 模型偶爾把回覆撞爆（酒款）

根本原因：草稿要模型輸出固定 14 欄的 JSON，但模型偶爾暴衝、把回覆寫爆 token 上限（finish_reason=length）→ JSON 寫到一半被截斷、沒有結尾 → 整份解析失敗。

→
解法：拉高 token 上限給緩衝 ＋ 解析改「逐欄位保底」（欄位固定，就算截斷也把斷點前的欄位一個個抠出來當草稿），不再是「整包成功或整包失敗」。原則：面對 AI 不穩定的輸出，讓解析能優雅降級，比要求它每次完美實際。
07

如果繼續往下

  • 喵哥從主管群開放到全員：問答免費，但若要全店每天推播提醒，得先評估 LINE 付費方案。
  • 事件通知：表單截止、班表發布、臨時徵班——可複用同一套 LINE 基礎建設。
  • 酒款接上銷售數據：把 POS 銷售接進來，喵哥就能回答「這支好不好賣」，選酒筆記也能長出真實回購紀錄。
  • 知識分層裁剪：酒款累積到上百支時，把售完的收成一行摘要，替 system 的知識段瘦身、控成本。
Takeaway

兩個模組、一套可複用的骨架：(1) 聊天走 LINE 免費 reply 通道，成本趨近於零；(2) 知識用「全量上下文 + prompt caching」而非 RAG，小規模下更準也好維護；(3) 用 OpenRouter 把模型變成可隨時抽換的字串；(4) 需要結構化輸出（酒款 JSON）或看圖辨識時，假設 AI 會出錯——圖沒送到就停、格式壞就逐欄位救、沒把握就回頭問人，別讓 AI 的錯直接落地。

能長這麼快的特殊條件是——排班系統的地基（LINE 帳號、員工資料、知識庫）本來就在。全新從零開始，前期得多花時間搭地基。
