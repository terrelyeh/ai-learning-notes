# 喵哥 — 把店裡的 GPTs 助理搬進 LINE：一個餐飲業主的第一支 AI 機器人

> 🤖 **我用 AI 做了什麼**：把原本只能在 ChatGPT 裡用的店內 GPTs 助理「喵哥」，整支搬進 LINE 官方帳號，變成員工隨時能問的機器人，還順手長出一套酒類資料庫和後台。
> ⏱ **沒有 AI 的話**：我不會寫程式。這功能對我來說等於「做不出來」——不然就是外包幾萬塊、等好幾週。這次是一個晚上到隔天，跟 AI 對話就完成了。
> ✅ **最終成果**：LINE「下午酒場 喵哥」上線，員工能問制度、SOP、選酒，能傳照片、請它出題；後台有酒款 AI 建檔、對話紀錄、AI 用量監控，聊天室下方還有一張和紙風的圖文選單。

> 我是台北一間下午酒場的老闆，這是我第一次做 LINE 機器人。這篇記的不是「怎麼寫 code」（那是 AI 做的），而是**我在過程中搞懂的架構和知識點**——給同樣想用 AI 做點東西、但不是工程師的人參考。

---

## 一、為什麼要做這件事

店裡本來就有一個 ChatGPT 的 GPTs 助理「喵哥」，員工可以問它酒水、出餐、服務 SOP。但有個致命的摩擦：**要用它，得先打開 ChatGPT、還要登入。** 現場忙起來根本不會有人這樣做。

而員工每天本來就掛在 LINE 上。所以問題其實很單純：**能不能把喵哥搬到 LINE，讓大家在原本就在用的地方直接問？**

一開始我只想著「搬過去」。但跟 AI 討論的過程中，範圍一路長大——因為 AI 幫我看到：這個排班系統本身已經有一大半基礎建設（LINE 官方帳號、員工綁定、公司制度知識庫）可以直接複用，搬進來反而比從零做一個 GPTs 替代品更省。於是它不只搬了家，還接上了店裡的真實資料。

## 二、最終樣貌

員工在 LINE 直接問，喵哥依店內知識回答（下圖是問「拉佛格 highball 配什麼」的實際對話）：

- **LINE 問答**：制度 / SOP / 選酒，回答有店內依據、不瞎掰；能看酒標和交接單照片；能出題練習
- **圖文選單**：聊天室下方四格（我的班表 / 公司制度 / 全店班表 / 問問喵哥），和紙木刻風
- **酒類資料庫**：後台拍酒標照 → AI 產一份完整選酒筆記草稿，我過目補價格就上架
- **後台**：喵哥設定（開關 / 白名單 / 換模型 / 看大家問什麼）、AI 用量監控

## 三、系統架構

搞懂這張圖，就懂整個機器人怎麼運作了：

```
員工在 LINE 傳訊息
      │
      ▼
LINE Messaging API webhook ──（傳 6 碼）──▶ 綁定流程（把 LINE 帳號對到員工）
      │（其他文字 / 圖片）
      ▼
查身份 ＋ 開關 / 白名單把關
      │
      ▼
組一份「問題包」：喵哥人設 ＋ 店內知識庫（整包）＋ 這個人最近的對話
      │                                    ▲
      ▼                                    │
   OpenRouter（大語言模型）          知識庫三入口：
      │                              制度文件 / 上傳檔案 / 酒類資料庫
      ▼
回覆員工（免費的「回覆」通道）＋ 把對話存起來（＝下次的記憶）
```

三個關鍵理解：
- **webhook** 就是「LINE 收到訊息時，自動打電話到我的系統」的那條線。機器人的一切從這裡開始。
- **知識庫是「每次都整包塞給 AI」**，不是 AI 去資料庫裡搜（這點很反直覺，見第五節）。
- **記憶是自己存的**——AI 本身沒有記憶，是我們把最近的對話撈出來、跟這次的問題一起交給它。

## 四、技術選型 + 外部服務與金鑰

| 工具 / 技術 | 選它的理由 |
|---|---|
| LINE Messaging API | 員工本來就在用 LINE；官方帳號的「回覆」訊息免費（見第五節），聊天機器人成本幾乎是零 |
| **OpenRouter** | 一個帳號、一把金鑰就能用所有家的模型（Claude / GPT / Gemini…），想換模型改一個字就好，不用分別去各家開帳號 |
| Claude Sonnet 5 | 起步的主力模型：中文語感和「人味」在這個用途最穩 |
| Supabase（既有） | 排班系統原本就在用的資料庫，酒庫、對話紀錄直接加表 |
| Next.js（既有） | 排班系統本體，後台頁面加在同一個專案 |
| nano banana（生圖）| 圖文選單的和紙風底圖——用 AI 生藝術、我再疊上精準中文（見踩坑四）|

**外部服務與金鑰**（要讓它跑起來，得先準備這些）：

| 服務 / 模型 | 用途 | 類型 |
|---|---|---|
| LINE 官方帳號「下午酒場 喵哥」 | 收發訊息、圖文選單 | Channel（Messaging API）|
| `LINE_CHANNEL_ACCESS_TOKEN` / `SECRET` | 呼叫 LINE API、驗證訊息來源 | Token / Secret |
| OpenRouter | 串接大語言模型 | AI 服務 |
| `OPENROUTER_API_KEY` | 每次呼叫 AI | API Key |
| `OPENROUTER_PROVISIONING_KEY` | 後台看餘額 / 用量 | API Key（管理用）|

> ⚠️ 一個實務提醒：OpenRouter 是**先儲值再用**的（我先充了 US$10 試水溫）。而且帳戶餘額是跟其他專案共用的，看用量時要認清楚哪一筆是喵哥花的。

## 五、我學到的關鍵知識點（架構層）

這節是這篇筆記的重點——四個讓我「原來如此」的認知。

### 1. LINE 有兩種訊息，一種免費、一種計費——這決定了成本

- **回覆訊息（reply）**：使用者傳來、機器人回應——**免費、無上限**。喵哥每一次回答都走這條，所以**再多人問、問幾百次，都是零成本**。
- **主動推播（push）**：機器人主動送的（例如每天的班表提醒）——**計費，而且逐人算**。對 20 個人各推一則就是 20 則。

免費版一個月 200 則額度，算的是 push。所以真正吃額度的不是聊天，是「每天主動提醒」：N 個人 × 30 天，大概 6~7 人就爆 200。**這個認知直接影響營運決策**（全店要收每日提醒就得升付費方案），而聊天完全不受限。

### 2. 知識庫是「全量上下文 + 快取」，不是 RAG

我以為 AI 機器人一定是那種「去資料庫裡搜出相關段落再回答」的架構（RAG）。但這個規模不需要——我們的做法是**每次對話，把店裡全部知識（制度、公告、酒單）整包塞進去給 AI**，靠一個叫 **prompt caching** 的機制讓重複的部分只算 1/10 的錢。

為什麼這樣反而好：
- AI **看得到全貌**，跨酒款比較、「客人喜歡 A 推什麼」這種題目不會因為「搜尋漏抓」而答錯
- 零檢索基礎建設，維護簡單

什麼時候要換：知識大到某個量（約十萬字級）成本和延遲才有感。那時第一步不是急著上 RAG，而是「**分層裁剪**」——現貨酒款給完整資料、售完的只留一行摘要。這比 RAG 簡單、又保住準確度。

### 3. 透過 OpenRouter，換模型變成「改一個字」

我原本以為要用哪家的 AI 就得去那家開帳號、綁卡、學它的接法。OpenRouter 是一個中間層：一把金鑰，背後接所有家。想從 Claude 換成 GPT 或 Gemini 比較效果，只要在後台改一個模型名稱。這讓「A/B 試哪個模型寫得好」變得幾乎沒有門檻——我甚至把「聊天用的模型」和「寫文案用的模型」分開設定。

### 4. AI 建檔要「人在迴路」，而且「上網查」要放對時機

酒類資料庫最聰明的一步：拍張酒標照 → AI 辨識 + 上網查 + 照店裡的筆記範本填好草稿 → **我過目、補上店內價格和出杯規格 → 才上架**。

關鍵決策是**把「上網查」放在建檔那一刻，而不是員工提問的當下**。因為：
- 小眾清酒的網路資料很零散、常是日文，AI 容易腦補——錯的話員工會直接講給客人聽
- 建檔時我會過目，錯的當場攔下；而且全店講法一致（不會員工 A、B 問同一支酒得到不同答案）

這就是「人在迴路（human-in-the-loop）」：AI 做粗活（查、寫草稿），人做最後把關。既省我時間，又不失控。

## 六、AI 怎麼幫我做的

**分工**：我負責所有「判斷」——要不要做、範圍多大、用哪個模型、叫什麼名字、風格對不對、上線後哪裡怪。AI 負責所有「執行」——架構設計、成本估算、全部程式、抓 bug、生圖、寫文件。我一行 code 都沒寫。

**我用了哪些提問方式**（回頭看蠻有意思的）：
- **範例驅動**：直接把店裡原本的 GPTs 提示詞貼給 AI 當「規格書」，說「照這個做但搬到 LINE」。有現成參考，AI 少走很多冤枉路。
- **症狀描述 + 截圖**：喵哥回答排版變糟時，我貼了「改之前 vs 改之後」兩張對話截圖，讓 AI 自己看出問題。比我用文字形容準得多。
- **探索式提問**：我會問「這是 RAG 嗎？」「什麼情況算一則訊息？」——這些「不是要它做事、是要它教我」的問題，反而是我學最多的地方。
- **決策式對齊**：碰到分岔（範圍、模型、風格），AI 會把選項和取捨列給我勾，而不是自己猜。

**三個關鍵轉折**：
1. **一句閒聊讓範圍升級**：本來只想搬 GPTs，我隨口問「說不定可以建店裡自己的酒類資料庫？」——AI 順著把它從「一份筆記文件」升級成有狀態、能篩選、會累積的**資料庫**。最好的功能常是聊出來的。
2. **先上線、再用真實對話回饋**：喵哥上線後我實際去問，發現排版變一坨、草稿會逾時、對話紀錄是空的。這些**只有真的用了才會冒出來**。一輪輪修，比想一次做到完好太多。
3. **對品牌的堅持**：第一版圖文選單我嫌「太像一般 App、跟店的調性不合」，要求用萬華世界的和紙木刻風。AI 就改用生圖模型產底圖、再疊上精準中文。**堅持品味，AI 有能力接住。**

## 七、踩到的坑，讓我更懂的事

（這些都是 AI 幫我診斷和修的，但每一個都讓我多懂一點）

### 坑一：對話紀錄一直是空的 → 原來「記憶」從沒生效
後台的對話紀錄一筆都沒有。查下去發現：**寫入資料庫時「靜默失敗」了**——程式沒報錯、但每次都沒寫進去，連帶喵哥的「多輪記憶」從上線起就沒作用（每句話都當新對話在答）。
> 帶走的原則：**「沒報錯」不等於「成功」。** 會影響資料的操作，一定要主動檢查有沒有真的寫進去，不然壞了都不知道。

### 坑二：喵哥回「有點忙不過來」，草稿也空白 → token 被「思考」吃光了
新一代 AI 有「深度思考」模式，回答前會在內部先想一長串——**而這些思考也算進回覆的字數額度**。某些線路預設把思考打開，結果額度被思考燒光、正文一個字都吐不出來。
> 帶走的原則：AI 的「回覆長度」是有預算的，而思考會偷吃這個預算。即答型的場景要把它關掉。

### 坑三：改了一句人設，喵哥講話變一坨
我想讓回答更簡潔，結果改壞了——喵哥把答案全擠成一段、不分行不列點。原因是我加的一句「短問題幾行講完」被模型過度執行。
> 帶走的原則：**提示詞是活的，牽一髮動全身。** 改完一定要用真實問題實測輸出再上，不能憑感覺盲改。（後來我們養成習慣：人設一改，先拿真問題跑一次看結果。）

### 坑四：要選單圖，但 AI 畫不出正確的中文字
想用 AI 生圖做圖文選單，但圖像模型畫多個中文字常常缺筆畫、糊掉。
> 帶走的原則：**圖像模型擅長「畫」、不擅長「精準的字」。** 解法是混合法——讓 AI 生藝術底圖，文字用程式精準疊上去，兩邊各做各的強項。

## 八、如果繼續往下

- **從主管群開放到全員**：目前先開給主管測試。全員之前要先算 LINE 額度——聊天免費，但如果之後要全店每天推播提醒，得評估付費方案。
- **事件通知**：表單截止、班表發布、臨時徵班——這些「主動通知」可以複用同一套 LINE 基礎建設。
- **知識變多時的分層裁剪**：酒款累積到上百支時，把售完的酒收成一行摘要，替 AI 的「問題包」瘦身。
- **接上銷售數據**：未來把 POS 的銷售資料接進來，喵哥就能回答「這支好不好賣」，選酒筆記也能長出真實回購紀錄。

## Takeaway

**這個案例展示了什麼**：一種「決策我來、執行 AI 來」的協作節奏，加上三個很好複製的習慣——(1) 把現有的東西（店裡的 GPTs 提示詞）直接當規格書餵給 AI，省掉大量來回；(2) 先上線、再用真實使用回饋迭代，而不是追求一次做到完美；(3) 用截圖描述症狀，比用文字形容問題精準得多。

**可移植性**：很適合「手上有明確參考範本、而且願意分批迭代」的專案。特殊條件是——我這個案子能長這麼快，是因為排班系統的地基（LINE 帳號、員工資料、知識庫）本來就在；如果是全新從零開始，前期會多花一些時間搭地基。

**如果你也要做，可以先問 AI**：
- 「我想把一個 ChatGPT GPTs 的助理搬到 LINE 官方帳號，背後串大語言模型。先幫我盤一下需要哪些元件、哪些可以用現成服務、大概的成本結構是什麼？」
- 「LINE 官方帳號的『回覆訊息』和『主動推播』在計費上差在哪？免費額度怎麼算？」——先搞懂成本結構，再決定功能範圍。
