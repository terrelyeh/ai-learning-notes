# 喵哥 — 把店裡的 GPTs 助理搬進 LINE：架構與技術筆記

> 🤖 **做了什麼**：把原本只能在 ChatGPT 裡用的店內 GPTs 助理「喵哥」，整支搬進 LINE 官方帳號，背後串大語言模型、接上店裡的制度／SOP／酒單知識，還長出一套酒類資料庫。
> ⏱ **沒有 AI 的話**：我不會寫程式，這功能等於「做不出來」——不然就是外包幾萬塊、等好幾週。這次跟 AI 對話一個晚上到隔天就上線了。
> ✅ **成果**：LINE「下午酒場 喵哥」上線——知識庫問答、看圖、酒款 AI 建檔、對話紀錄與 AI 用量監控。

> 我是台北一間下午酒場的老闆、不是工程師。但這篇不寫「怎麼跟 AI 聊天」，而是把這支 LINE 機器人**實際怎麼運作**講清楚——訊息怎麼進來、知識怎麼組成 prompt、模型怎麼呼叫、一則訊息怎麼算錢、哪些地方會壞。code 是 AI 寫的，架構是我搞懂的。

---

## 一、為什麼要做

店裡本來就有一個 ChatGPT 的 GPTs 助理「喵哥」，員工可以問它酒水、出餐、服務 SOP。但有個致命摩擦：**要用它得先打開 ChatGPT、還要登入。** 現場忙起來沒人會這樣做。而員工每天本來就掛在 LINE 上。

所以問題很單純：**把喵哥搬到 LINE，讓大家在原本就在用的地方直接問。**

一開始只想「搬過去」，但範圍很快長大——因為排班系統本身已經有一半基礎建設可以直接複用：**同一個 LINE 官方帳號**（原本做班表提醒用）、**員工綁定關係**、**公司制度知識庫**。與其從零做一個 GPTs 替代品，不如接進這套既有系統，順便讓喵哥吃到店裡的真實資料。這個「複用既有地基」的判斷，是它能一個晚上長出來的關鍵。

## 二、系統架構

整支機器人就是一條「訊息進 → 組 prompt → 問模型 → 回覆」的管線：

```
員工在 LINE 傳訊息
      │  LINE 平台 POST 到我的 webhook
      ▼
/api/line/webhook  ──①驗簽（X-Line-Signature）
      │
      ├─ 純 6 碼數字 ──▶ 綁定流程（把 LINE 帳號對到員工）
      │
      └─ 其他文字／圖片 ──▶ ②查身份 ＋ 開關／白名單把關
                                   │
                                   ▼
                          ③組 prompt（system：人設＋知識庫整包
                                       user：發問者＋這句話）
                                   ＋ 撈近 6 小時對話當記憶
                                   │
                                   ▼
                          ④OpenRouter → 大語言模型
                                   │
                                   ▼
                          ⑤用 replyToken 回覆（免費）＋把這輪對話寫進 DB
```

拆開來看五個技術點：

**① webhook 驗簽**：LINE 收到員工訊息後，會 POST 到我指定的網址（webhook）。但任何人都能打這個網址，所以每個請求都帶一個 `X-Line-Signature`——用我的 channel secret 對內容算 HMAC-SHA256，對不上就丟掉。這是「確定這則訊息真的是 LINE 送來的」的把關。

**② 身份與權限**：webhook 先看內容——**純 6 碼數字**走綁定（把發訊息的 LINE 帳號寫進對應員工），其他文字／圖片才進問答。進問答前先過兩道閘：總開關 `bot_enabled`、白名單 `bot_allowlist`（目前只開主管群）。

**③ 組 prompt**：這是最核心的一步。把三樣東西組成一次 API 呼叫的訊息陣列——
- `system` 段：**喵哥人設** ＋ **店內知識庫（整包純文字）**。知識庫有三個入口（制度文件 / 上傳檔案 / 酒類資料庫），但都先匯進同一條組裝管線變成一段文字。
- `user` 段：發問者身份 ＋ 這次的問題。（身份放 user 而不是 system，是為了不污染快取，見第四節。）
- **記憶**：撈這個員工近 6 小時的對話（上限約 12 則）一起帶上——大語言模型本身沒有記憶，「記得剛剛講什麼」是我們自己去 DB 撈出來喂回去的。

**④ 呼叫模型**：透過 OpenRouter 送給大語言模型，幾個關鍵參數：關掉模型的內部「思考」（見踩坑二）、需要時開 web 搜尋外掛、Anthropic 系模型在知識段標記快取。

**⑤ 回覆＋存檔**：用 LINE 的 `replyToken` 回覆（**免費**，見第四節），同時把「使用者這句 + 喵哥這句」寫進 `bot_messages`——這既是給主管看的紀錄，也是**下一輪的記憶來源**。

## 三、技術選型與外部金鑰

| 工具 / 技術 | 選它的理由 |
|---|---|
| LINE Messaging API | 員工本來就在用；官方帳號的「回覆」訊息免費（見第四節），聊天成本幾乎是零 |

## 四、關鍵技術點

### 1. LINE 有兩種訊息：一種免費、一種計費——這決定成本結構

- **回覆訊息（reply）**：使用者傳來、機器人用 `replyToken` 回應——**免費、無上限**。喵哥每次回答都走這條，所以再多人問、問幾百次都是零成本。
- **主動推播（push）**：機器人主動送的（例如每天班表提醒）——**計費、逐人算**，對 20 人各推一則就是 20 則。

免費版一個月 200 則額度算的是 push。真正吃額度的不是聊天，是「每天主動提醒」：N 人 × 30 天，大約 6~7 人就爆 200。**這個認知直接決定功能邊界**——問答可以全店免費開，但全店每日推播提醒得評估付費方案。

### 2. 知識庫是「全量上下文 + prompt caching」，不是 RAG

我以為 AI 機器人一定是「去資料庫搜相關段落再回答」（RAG）。但這個規模不需要——做法是**每次對話把店裡全部知識（制度、公告、現貨酒單）組成一段純文字，整包塞進 system**。

省錢靠的是 **prompt caching**：把不變的知識段標記起來，模型第一次讀算全額、之後重複的部分只算約 1/10。要吃到快取有個前提——**那段內容必須逐字節穩定**（所以「發問者是誰」這種每次會變的東西要放到 user 段，不能混進被快取的 system 段，否則快取每次失效）。

這樣做的好處：模型**看得到全貌**，「客人喜歡 A 該推什麼」這種跨酒款比較不會因為「搜尋漏抓」而答錯；而且零檢索基礎建設、維護簡單。什麼時候該換：知識大到約十萬字級，成本與延遲才有感。那時第一步不是上 RAG，而是**分層裁剪**——現貨酒款給完整資料、售完的只留一行摘要。比 RAG 簡單又保住準確度。

### 3. 用 OpenRouter 做「模型路由」，換模型是改一個字串

OpenRouter 是一個中間層：一把金鑰、背後接所有家模型，介面統一（OpenAI 相容格式）。想從 Claude 換成 GPT 或 Gemini 比效果，只要改一個模型名稱字串。這讓 A/B 幾乎沒有門檻——我甚至把**聊天模型**和**寫酒款文案的模型**分成兩個設定（`bot_model` / `bot_draft_model`），聊天要人味、文案可以挑更會查資料的那顆。

### 4. AI 建檔：human-in-the-loop，而且「上網查」要放對時機

酒類資料庫的建檔流程：**拍酒標照 → AI 辨識＋上網查＋照店裡範本填成 JSON 草稿 → 我過目、補店內價格與出杯規格 → 才上架**。

關鍵決策是**把「上網查」放在建檔那一刻，而不是員工提問的當下**：小眾清酒的網路資料零散、常是日文，模型容易腦補；放在建檔時，錯的我當場攔下、而且全店講法一致（不會員工 A、B 問同一支酒得到不同答案）。這就是 human-in-the-loop——AI 做粗活（辨識、查、寫草稿），人做最後把關，既省時間又不失控。

## 五、踩到的坑（都是技術問題）

### 坑一：對話紀錄一直是空的 → 多輪記憶從沒生效
**症狀**：後台對話紀錄一筆都沒有，喵哥每句話都當新對話在答。
**根本原因**：寫入 DB 時把「使用者這句」和「喵哥那句」兩列一起批次 insert，但兩列的欄位組不一致，被資料庫的 API 層（PostgREST）整批拒絕（`PGRST102`）——而程式**沒有檢查回傳的錯誤**，於是靜默失敗。連帶記憶（撈近 6h 對話）一直撈到空的。
**解法**：兩列補齊成相同欄位組（缺的補 null）＋ 寫入後檢查 error。原則：**「沒報錯」不等於「成功」**，會改資料的操作一定要驗證真的寫進去了。

### 坑二：喵哥回「有點忙不過來」、酒款草稿也空白 → token 被「思考」燒光
**症狀**：喵哥不回答、草稿產出來是空的。
**根本原因**：新一代模型有「深度思考（reasoning）」模式，回答前會在內部先想一長串——**這些思考也算進回覆的 token 額度**。某些供應商預設把 reasoning 打開，結果額度被思考燒光、正文一個字都吐不出來（`finish_reason=length`）。
**解法**：即答型場景一律帶 `reasoning:{enabled:false}` 關掉。原則：模型的「回覆長度」是有預算的，思考會偷吃這個預算。

### 坑三：改一句人設，喵哥講話擠成一坨
**症狀**：想讓回答更簡潔，結果喵哥把答案全擠成一段、不分行不列點。
**根本原因**：我加的一句「短問題幾行講完」被模型過度執行。
**解法**：提示詞牽一髮動全身，改完**一定要拿真實問題實測輸出再上**，不能憑感覺盲改（後來養成習慣：人設一改先跑真問題看結果）。

### 坑四：要選單圖，但圖像模型畫不出正確中文
**症狀**：用 AI 生圖做圖文選單，但圖像模型畫多個中文字常缺筆畫、糊掉。
**根本原因**：圖像模型擅長「畫」、不擅長「精準的字」。
**解法**：混合法——讓 AI 生藝術底圖，文字用程式精準疊上去，兩邊各做各的強項。下面的 Before / After 就是這樣做出來的。

### 坑五：酒款 AI 草稿「時好時壞」噴解析錯誤 → 模型偶爾暴衝把回覆撞爆
**症狀**：同一張酒標，AI 建檔時好時壞，偶爾跳 `JSON 解析失敗`。
**根本原因**：草稿要求模型輸出一份 JSON（店裡固定 14 個欄位），但模型**偶爾暴衝**、把回覆寫爆 token 上限（`finish_reason=length`）→ JSON 寫到一半被截斷、沒有結尾 → 整份解析失敗。清楚的酒標很少中，模糊／壓縮過的圖較容易觸發。
**解法**：三層防護——① 拉高 token 上限給緩衝（正常用不到、只在暴衝時救得回完整）；② 解析改「逐欄位保底」：欄位是固定的，就算截斷也把斷點前的欄位一個個抠出來當草稿，不再是「整包成功或整包失敗」；③ prompt 強制「一定輸出 JSON、不要用純文字回問」。原則：**面對 AI 不穩定的輸出，與其要求它每次完美，不如讓解析能優雅降級。**

## 六、如果繼續往下

- **從主管群開放到全員**：目前 `bot_allowlist` 只開主管測試。全員之前要先算 LINE 額度——問答免費，但若之後要全店每天推播提醒，得評估付費方案。
- **事件通知**：表單截止、班表發布、臨時徵班——這些主動通知可複用同一套 LINE 基礎建設（綁定、push、驗簽都現成）。
- **知識分層裁剪**：酒款累積到上百支時，把售完的酒收成一行摘要，替 system 的知識段瘦身，控制成本與延遲。
- **接上銷售數據**：把 POS 銷售資料接進來，喵哥就能回答「這支好不好賣」，選酒筆記也能長出真實回購紀錄。

## Takeaway

這支機器人可複用的技術骨架，其實就四塊：**(1)** 聊天走 LINE 免費的 reply 通道、把成本壓在趨近於零；**(2)** 知識用「全量上下文 + prompt caching」而不是 RAG，小規模下更準也更好維護；**(3)** 用 OpenRouter 把模型變成可隨時抽換的字串；**(4)** 需要結構化輸出（酒款 JSON）時，讓解析能對截斷／髒格式優雅降級，而不是要求模型每次完美。

能長這麼快的特殊條件是——排班系統的地基（LINE 帳號、員工資料、知識庫）本來就在。全新從零開始的話，前期得多花時間搭這些地基。
