工具開發 · AI 協作

愛嬌冰店 BEP 試算器

開冰店之前，團隊最需要回答的問題是：「每個月至少要賣多少，才不會虧錢？」這個工具就是為了回答這個問題而生。

開發期間：2026-02 至 2026-03  ·  類型：工具開發 + AI 協作
我用 AI 做了什麼 從零建立一個互動式損益平衡分析工具，讓冰店經營團隊能即時試算固定成本、BEP、目標利潤，並模擬 B2B 通路效益
沒有 AI 的話 需要找工程師開發或用 Excel，互動性和視覺化程度會大幅降低，估計 2-3 週
最終成果 一個純 HTML 單檔的互動式計算器，部署在 Cloudflare Pages，團隊用瀏覽器就能操作
01

為什麼要做這件事

愛嬌冰店在籌備階段，團隊需要一個共同的數據基礎來討論營運策略。傳統做法是開 Excel 試算表，但 Excel 有幾個問題：

  • 不直觀：非財務背景的夥伴很難快速理解數字之間的關係
  • 不互動：改一個參數要手動更新一堆儲存格，容易出錯
  • 不好溝通：開會時很難即時調參數給大家看

真正的需求不是「做一個計算機」，而是讓團隊在同一個畫面上，即時看到不同決策對損益的影響。

02

技術選型

Tailwind CSS Chart.js 4.4 Pure HTML Cloudflare Pages Inter + Noto Sans TC
工具選擇理由
純 HTML 單檔不需要任何建置步驟，用瀏覽器開就能跑，任何人都能使用
Tailwind CSS（CDN）快速做出好看的 UI，不需要寫 CSS 檔案
Chart.js 4.4（CDN）輕量的圖表庫，做損益敏感度折線圖剛好
Inter + Noto Sans TCInter 的數字清晰銳利（tabular-nums），中文用 Noto Sans TC
Cloudflare Pages免費部署，push to GitHub 自動上線

選型原則：能不加的依賴就不加。整個應用就是一個 index.html，沒有 package.json、沒有 build pipeline、沒有 node_modules。任何人拿到檔案，雙擊就能用。
03

推進方式

整個開發過程是一輪一輪的迭代，每一輪都是「我說需求 → AI 提方案 → 我看結果給反饋 → AI 調整」。

第一階段：建立基礎計算模型

先把最核心的東西做出來：產品售價 × 毛利率、固定成本明細、BEP 公式。左邊調參數，右邊即時算結果。關鍵判斷是「哪些參數要讓使用者調、哪些寫死」— 我決定讓所有成本項目都可調，因為團隊還在探索階段。

第二階段：加入情境比較

團隊會問：「如果只賣霜淇淋呢？如果聖代佔比多一點呢？」所以加了情境切換（全霜淇淋 / 各半 / 全聖代 / 自訂比例），讓不同產品組合的銷量目標可以直接對照。

第三階段：B2B 酒場通路模組

愛嬌有機會透過酒場通路銷售，但酒場要抽 20% 毛利作為返利。這個模組用 toggle 開關控制，開啟後才出現相關的輸入欄和計算結果。是一個完全選用的功能。

第四階段：從「打平」到「賺錢」

做完前三階段後，我意識到 BEP 只回答了「不虧錢要多少」，但團隊真正要規劃的是盈利策略。所以加了目標利潤模式（反推需要的營業額）和變動成本率（把包材、行銷等費用納入計算）。這兩個功能向下相容 — 設為 0 時結果跟原來完全一樣。

第五階段：UI 打磨

數字全部換 Inter 字體、結果卡片加立體陰影、參考區間做成可收合、Reset 按鈕放到卡片右上角。這些都是實際使用時覺得「不太對」然後一個一個調整的。

04

踩到的坑，讓我更懂的事

⚠ BEP 只算固定成本，數字會偏樂觀

一開始的模型只考慮固定成本（房租、薪資、折舊），算出來的 BEP 看起來很容易達成。但實際上每賣一份都有包材、金流手續費等變動成本，這些會侵蝕毛利。後來加了變動成本率，BEP 瞬間跳高不少。

→
財務模型的第一版永遠是不完整的。先做出來，用了之後才知道少了什麼。
⚠ 功能加了但使用者不知道在哪

目標利潤的 Reset 按鈕一開始放在輸入欄旁邊，我自己用的時候完全忽略它。移到卡片右上角、加上圖示之後才覺得自然。同樣的，變動成本的參考區間改成可收合之後，整個左欄清爽很多。

→
功能做出來只完成一半，放在對的位置、用對的方式呈現才是另一半。
⚠ 酒場通路的計算邏輯要獨立

B2B 通路的毛利計算（扣完返利後的淨毛利）不能跟直營的 BEP 公式混在一起。酒場貢獻是先從固定成本分子扣除，變動成本率只作用在直營的分母。搞混的話數字會莫名對不上。

→
當模型有多條通路時，每條通路的經濟模型要分開算，最後才合併。
05

AI 協作分工

這個專案幾乎所有程式碼都是 AI 寫的，但所有商業邏輯和設計判斷都是我做的。

AI 負責
  • 所有 HTML / CSS / JavaScript 程式碼
  • Chart.js 圖表設定與漸層效果
  • 公式邏輯的實作（BEP、混合毛利率、B2B 貢獻）
  • UI 元件的切版與排版
  • 提出功能優化建議
我負責
  • 定義核心目的與商業邏輯
  • 所有財務參數的判斷（返利比例、勞保算法等）
  • UI 使用體驗反饋與方向修正
  • 決定功能的優先順序
  • 審核每次改動是否符合業務邏輯
→
最有效的協作模式：我先說「想要什麼效果」，AI 先說方案而不是直接動手，我確認方向對了再讓它執行。這避免了做完才發現方向錯的浪費。
06

最終樣貌

一個純 HTML 單檔的互動式損益平衡分析工具，包含：

  • 左側 5 張控制卡片（產品、通路、營運、變動成本、固定成本）
  • 右側即時更新的結果區（BEP / 目標營業額、銷量目標、敏感度圖、成本結構圖）
  • 酒場 B2B 通路模擬（選用，toggle 開關）
  • 密碼閘門保護，部署在 Cloudflare Pages
07

如果繼續往下

  • 加入「參數快照」功能，讓團隊可以存兩三組方案並排比較（保守版 vs 樂觀版）
  • 做一個「匯出為圖片」的按鈕，方便截圖放進簡報或傳到 LINE 群組
  • 在敏感度圖上加一條可拖拉的「預估營業額」標記線，直觀看到離目標還差多少
  • 等實際營運數據進來後，加入「實際 vs 預估」的對比功能
08

Takeaway

這個案例展示了什麼

非工程師也能用 AI 從零建立一個完整的互動式財務分析工具。關鍵不是會不會寫程式，而是能不能清楚定義「這個工具要回答什麼問題」以及「使用者會怎麼操作」。

可移植性

這個方法適用於任何「需要一個互動式試算器來輔助團隊決策」的場景 — 不只是餐飲業的 BEP，也可以是定價策略模擬、人力成本試算、廣告 ROI 計算等。純 HTML 單檔的架構適合不需要後端的靜態計算場景。如果需要存取資料庫或多人即時協作，就不適合這個架構了。

如果你也要做，先問 AI 什麼

1.「我想做一個互動式的 [X] 試算器，使用者可以調整 [參數 A、B、C]，即時看到 [結果 X、Y、Z]。請用純 HTML 單檔實作，不要用任何框架。先告訴我你打算怎麼設計畫面佈局和計算邏輯，確認後再開始寫 code。」

2.「這個計算模型目前只考慮了 [固定成本]，但實際上還有 [變動成本] 會影響結果。你覺得應該怎麼把它加進公式裡？對現有的 UI 和計算邏輯影響大嗎？」
