# 名廚翻譯工作台 — AI 協作開發紀錄

> 🤖 **我用 AI 做了什麼**：用 3 天、純對話協作，幫客戶「名廚 MINGCHU」做出一個中⇄韓文章翻譯工作台——抓官網文章 → AI 逐段翻譯 → 左右對照編輯 → 虛擬總編二審 → 定稿，全程套用團隊自己的術語庫與編輯口吻。
> ⏱ **沒有 AI 的話**：我不寫程式。這種「前端 + 多模型 LLM + Google Sheets 後端 + 串流 + 部署」的東西，外包至少兩三週、好幾萬塊；自學更不可能。
> ✅ **最終成果**：功能完整、已上線（mingchu-tools.vercel.app）。編輯可以全程不懂韓文，靠「AI 反查原文」也能維護術語庫；翻完還能請一個「在地總編」二審挑漏譯、統一術語。

> 這個工具解決一個很具體的編輯痛點：名廚是以廚師為核心的飲食媒體，官網有中文也有韓文文章，常需要互譯。但丟給 ChatGPT 翻，每次譯名都不一樣（同一個主廚名、餐廳名、菜名翻得飄來飄去），而且譯者不一定看得懂另一邊的語言。這個工具把「術語一致 + 編輯口吻 + 人機協作」綁成一條龍。

---

## 一、為什麼要做這件事

驅動我的不是「想做個翻譯功能」，而是三個真實的編輯痛點：

1. **術語會飄**：丟通用 AI 翻，同一個概念這段一個譯法、那段另一個譯法。對媒體來說，主廚名、餐廳名、菜名譯不一致是大忌。
2. **譯者語言不對等**：負責中文的編輯不一定讀得懂韓文。看到一個翻得怪的中文詞想改、想收進術語庫，卻不知道原文是哪幾個字。
3. **沒有「定稿前的關卡」**：AI 翻完直接交，沒有一道「在地母語編輯」幫忙抓漏譯、去翻譯腔、統一用詞。

我已經幫同一個客戶做過營收儀表板，驗證過「Next.js + Google Sheets + Vercel」這套，所以這次直接沿用——**團隊的術語庫、規則、紀錄全部存在他們自己看得到的 Google Sheet**，在網頁改或在 Sheet 改都同步。這是關鍵設計：工具不是黑盒子，資料權還在客戶手上。

## 二、最終樣貌

一條龍的編輯流程，全部在一個頁面內：

- **抓取／貼文字**：貼 mingchu.io 網址自動抓內文轉 Markdown，或直接貼文字；**自動辨識語言方向**（有韓文→韓翻中，否則中翻韓）。
- **逐段串流翻譯**：整篇一次翻（保全文上下文）、邊翻邊一段段冒出來；自動套用術語庫、翻譯規則、編輯口吻；角色感知（標題意譯、內文忠實、專名不譯）。
- **左右對照編輯**：原文／譯文並排，點譯文即可改；偵測到的名廚用語會高亮。
- **AI 反查原文**：編輯改好一個中文詞想加進術語庫，但不懂韓文——選起來按「AI 找原文」，AI 從對齊的原文裡把對應的韓文片段挑出來，自動填好；對應詞在左側原文用淡綠底標出。
- **虛擬總編二審**：翻完按「總編審核」，由目標語言的在地總編逐段檢查——抓漏譯、誤譯、統一術語、去翻譯腔，給「**建議＋理由**」讓你逐段套用或忽略（建議還能先改再套用）。
- **定稿／複製**：定稿後寫回 Google Sheet 紀錄，可複製 Markdown 或格式化 HTML 貼回後台。

旁邊還有術語庫、規則、編輯方針、審核提示詞四個分頁，全部可在網頁維護、即時寫回 Sheet。

## 三、系統架構

整條流程「什麼跟什麼串在一起」：

```
  抓取 mingchu.io ─┐
  （cheerio解析）   │
                   ├─→ 自動辨識方向 ─→ 翻譯 API ─（NDJSON 串流）→ 逐段對照編輯
  貼文字 ──────────┘                  │  ↑
                                      │  └─ 術語庫 / 規則 / 編輯方針（Google Sheet 即時讀）
                                      │
  選字加術語 ←──────────────────────────┘
   └─→ AI 反查原文（map-term API）→ 自動填原文詞 → 寫回術語庫(Sheet)

  按「總編審核」─→ 審核 API（分批並行 / 獨立審稿模型）─→ 逐段建議 → 人工套用
                                      │  ↑
                                      │  └─ 審核提示詞（Sheet，可覆蓋內建）
  定稿 ─→ 寫回 Translations 分頁(Sheet)
```

**核心：所有資料都在 Google Sheet。** LLM 全部走 OpenRouter（一支 API、多家模型）。前端一律透過自家後端 route 呼叫，金鑰不進前端。部署在 Vercel，推 GitHub 自動上線。

## 四、技術選型

| 工具 / 技術 | 選的理由 |
|---|---|
| Next.js 16 + React 19 + Tailwind | 沿用我同客戶營收儀表板驗證過的那套，AI 最熟、踩坑最少 |
| Google Sheets 當資料庫 | 沿用既有 Service Account 與權限模型；**團隊在網頁或 Sheet 改都同步**，資料權留在客戶 |
| OpenRouter | 一支 API 接 GPT / Claude / Gemini 全家，換模型只換字串、不用各接各的 SDK |
| Vercel | 推 GitHub 自動部署，金鑰放環境變數 |

> **選型原則**：在 AI 最熟悉的範圍內選，不選最潮的。陌生技術＝每個 bug 都是地獄。能沿用就沿用。

## 六、AI 怎麼幫我做的

**分工**：我定方向（要做什麼功能、UX 怎麼走、模型取捨的原則），AI 負責全部實作、實測、除錯。我一行 code 都沒寫——我做的是**產品決策**和**提供真實使用情境**。

**提問模式**，我大概用了三種：

- **症狀描述型**：「翻一半就停住」「審核跳 Unexpected token」——把現象丟給 AI，讓它去查根因，而不是我猜。
- **決策諮詢型**：「中翻韓預設要不要換模型？」——要求 AI **先拿真實文章實測、用數據說話**，再一起拍板，不憑感覺。
- **迭代回饋型**：看了成品提下一個優化點（「字太小」「按鈕不明顯」「收合鍵位置怪」），一波一波打磨。

**三個關鍵轉折**（讓任務真正推進的時刻）：

1. **逾時的根因不是「模型慢」**。長文審核一直壞，一開始以為是 Vercel 60 秒上限把慢模型砍了。後來叫 AI 直接拿客戶那篇真文章去打 OpenRouter、看模型的**原始輸出**，才發現真兇是——**Sonnet 對「韓文輸出」根本不守 JSON 格式**（會在前面加韓文前言、用清單回）。這個坑用合成測資完全測不出來。

2. **沒有一個模型兩個方向都最好**。叫 AI 拿真文章做中翻韓的品質 PK，發現 GPT-5.1 中文在地用詞準（托斯卡尼）但韓文會出包（把「五柳」翻錯成「五味」）；GPT-5.2 反過來。於是決定**方向感知預設**（中翻韓用 5.2、韓翻中用 5.1），並把在地譯名鎖進術語庫。

3. **架構是被真實限制逼出來的**。虛擬總編審核從「單次呼叫」→「只回要改的（slim）」→「分批並行」，每一步都是被 Vercel 60 秒上限逼著演進，不是一開始就設計好的。

## 七、踩到的坑，讓我更懂的事

### 坑一：長文一翻就「停在一半」

- **症狀**：抓了長文按翻譯，邊翻邊出，但翻到一半就不動了，也沒報錯。
- **關鍵認知**：整篇一次翻＝輸出越多越久；慢模型的長文會超過 Vercel 60 秒被硬砍，而前端「串流斷了卻沒收到結束訊號」當時是**靜默停掉**。
- 解法是多層的：先上**串流**（邊翻邊看，體感快很多）、斷線改成**明確報錯不再靜默**、預設換成快模型。

> 帶走的原則：**「沒反應」不一定是壞掉，可能是它還在跑、或是它被環境（平台逾時）砍了而你沒接到。** 先把「失敗」變得看得見，再談修。

### 坑二：同一個模型，換個語言方向就壞掉

- **症狀**：中翻韓的審核一直回「格式異常」或逾時，但韓翻中完全沒事。
- **根本原因**：審核要模型回結構化 JSON。Sonnet 對**中文輸出**很乖，對**韓文輸出**就不守規矩（加前言、用 prose）。問題不在「方向」，在「**輸出語言**」。
- **解法**：審稿預設換成守 JSON 又快的 GPT-5.1；JSON 抽取也改得更穩（抓第一個 `{` 到最後一個 `}`）。

> 帶走的原則：**model 的「行為」會因任務、語言、版本而不同，而且這種坑合成測資測不出來——一定要拿使用者的真實內容下去打。** 我前面好幾次「合成測試看起來好、真文章一試就壞」，就是這個教訓。

### 坑三：與其追「完美模型」，不如鎖進術語庫

- 模型各有強弱方向，今天最強的下個月可能變。
- **認知**：托斯卡尼、五柳這種「在地譯名」，**用術語庫鎖死（每次都套同一個）比每次叫 AI 重判（會飄）更可靠**。這正是術語庫存在的意義。

### 坑四（介面）：「感覺太小」往往是實作偷偷縮水

- 做完功能進入 UI 打磨，使用者覺得字太小、區塊標題不明顯。
- **認知**：去翻設計規範才發現——**規範本來就寫了該多大，但實作全面比規範小了 1~2 級**。「優化」其實是「把實作拉回自己訂的規範」，不是重新設計。

## 八、Takeaway

**這個案例展示的 AI 使用方式**：把 AI 當「實作 + 實測 + 除錯」的夥伴，自己守住「產品決策 + 真實情境」這兩塊。最有效的兩個動作是——遇到問題用**症狀描述**讓 AI 查根因；要做選擇時，要求 AI **拿真實資料實測、用數據說話**，而不是大家一起憑感覺猜。

**可移植性**：「拿真實內容測、用數據決策」適用於任何 LLM 應用開發——模型選型、prompt 調校、延遲與穩定性問題都通用。要注意的特殊條件：Vercel 60 秒上限是 Hobby 方案特有的；模型的行為差異會隨版本變動，今天的結論半年後要重驗。

**如果你也要做，先問 AI 這兩句**：

1. 「這個功能在 ___ 情境下會壞，幫我**拿真實資料重現**，並去看模型/伺服器的原始回應查根因，不要用合成測資。」
2. 「同一個任務用 A／B／C 模型各跑一次，給我**速度＋品質＋成本**的對照表再建議，我要看數據才決定。」
