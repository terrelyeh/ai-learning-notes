# MINGCHU 營收儀表板 — AI 協作開發紀錄

> 🤖 **我用 AI 做了什麼**：把營收追蹤的 Google Sheet 變成有篩選、樞紐分析、編輯、獎金結算的互動式儀表板
> ⏱ **沒有 AI 的話**：找開發團隊估計 2-3 個月、預算數十萬，需求變動每次都要重新溝通規格
> ✅ **最終成果**：儀表板已上線（[mingchu-revenue.vercel.app](https://mingchu-revenue.vercel.app/)），團隊每週用，獎金結算流程從手動算 4 小時降到 5 分鐘

> 一個 8 人小團隊的營收儀表板，從業績追蹤、Pipeline 分析到結案後獎金分配，全流程在一個介面完成。

---

## 一、為什麼要做這件事

MINGCHU 是一間 8-10 人的行銷公司，主要服務食品/餐飲品牌客戶。原本的工作流程是：

- 業務在 Google Sheet 上維護「YYYY 營收」清單，記錄每個專案的金額、客戶、階段
- 每月、每季開會前，都要花時間手動 SUMIF / pivot 整理數字
- 結案後總經理要對 Sheet 算獎金（毛利率、池子計算、誰拿多少）

**痛點**：
- 開會前都要花時間整理報表
- 同一份資料每個人想看的角度都不一樣（按客戶、按業務、按行銷、按類別）
- 沒有專案編號，多 PM 合作時績效歸屬不清楚
- 獎金計算規則複雜，每次結算總經理都要重新對 Sheet
- 資料容易打錯（PM 名字、類別等都是手填）

**為什麼不用現成工具**：
- Salesforce 太貴、不適合這種規模
- 員工已經習慣 Google Sheet 編輯，換工具阻力大
- 需求很客製（行銷比例、結案獎金規則）

所以目標清楚：**Sheet 留著當資料源，做一個介面把各種角度的分析、編輯、獎金結算包進去**。

## 二、最終樣貌

主儀表板（[mingchu-revenue.vercel.app](https://mingchu-revenue.vercel.app/)）：

- **業績總覽 Tab** — 4 張 KPI 卡片（年度營收/現金/Pipeline/待收款）+ 季度進度條 + 月度趨勢圖
- **專案管理 Tab** — 5 種視角切換（列表 / 按客戶 / 業務 / 行銷 / 類別）、多維篩選、搜尋、欄位顯示切換
- **多年趨勢頁** — 歷年營收堆疊圖 + YoY + 客戶 YoY 表
- **獎金結算頁** — 待分配 / 已分配 / 不符合 三 tab，毛利率審核 + 獎金池計算 + 人員分配

亮點功能：
1. **多行銷制** — 一案最多 3 位行銷各自比例分配，Modal 動態加減
2. **專案編號** — A 欄系統自動產生 `P26-001`、`P26-002`...，讓識別碼跟列號解耦
3. **URL 同步** — 切換視角/篩選都會記在網址，可直接複製分享給同事
4. **權限分層** — 一般員工看儀表板，總經理才能看獎金結算

## 三、系統架構

```
Google Sheet（6 個 tabs）
  ├── YYYY 營收           ← 專案明細（每年一個 tab）
  ├── 目標設定            ← 年度 L/H 目標
  ├── 歷年營收            ← 各年度行銷+其他營收
  ├── 人員設定            ← 所有人員 + 角色 + 在職
  ├── 選項設定            ← 階段 + 類別 master list
  └── 獎金分配            ← 系統自動寫入
        ↓
Sheets API（Service Account 認證）
        ↓
Next.js API Routes（Vercel Serverless）
  ├── /api/auth/verify    Google ID Token + email 白名單
  ├── /api/revenue        GET 讀取（含編號自動 backfill）
  ├── /api/revenue/update POST 寫回 Sheet
  └── /api/bonus          GET/POST 獎金分配（admin only）
        ↓
Frontend（Next.js + React）
  ├── /                   主儀表板（業績 + 專案管理）
  ├── /trends             多年趨勢頁
  ├── /bonus              獎金結算頁
  └── /guide.html         使用指南（靜態）
        ↓
Vercel（自動部署 + 環境變數管金鑰）
```

關鍵架構決策：
- **Google Sheet 當資料庫**，不接 Supabase / MySQL（資料量小、團隊熟悉、零搬遷成本）
- **Service Account 共享 Sheet**，前端永遠透過後端 API 讀寫（不曝露金鑰）
- **權限三層**：能登入（`ALLOWED_EMAILS`）→ 能看大部分頁面 → 能看獎金結算（`BONUS_ADMIN_EMAILS`）

## 四、技術選型

| 工具 / 技術 | 選的理由 |
|---|---|
| **Next.js 15 + React 19** | 一套搞定前後端，AI 最熟悉的堆疊 |
| **shadcn/ui (base-ui 版)** | 元件夠用、能客製、樣式不會太「模板感」 |
| **Tailwind CSS** | 沒設計稿也能寫得快、看得過去 |
| **Recharts** | React 生態最成熟的圖表庫 |
| **Google Sheets API v4** | 資料就在 Sheet，零搬遷成本 |
| **Google Identity Services** | 員工都有 Google 帳號，登入體驗好 |
| **Vercel** | 推 GitHub 就部署，環境變數好管 |

> **選型原則**：在 AI 最熟悉的範圍內選，不是選「最潮」的。陌生技術 = 每個 bug 都是地獄。

## 六、推進方式

整個專案不是一次規劃完，而是**根據實際使用 + 規則演化逐步加功能**。重要的推進策略：

**1. 從 Sheet 結構推回來**
每加一個功能，都先問：Sheet 上要怎麼存？欄位該長什麼樣？這個邏輯讓資料源永遠是「正確答案」，UI 只是 view layer。

**2. 真實規則出現時才實作**
獎金結算的規則一開始我以為是「固定率 × 個人貢獻」，做到一半才知道實際規則是「結案 + 毛利率 > 30% → 池子 = 5%、總經理主觀分配」。直接重做，沒堅持沿用舊設計。

**3. 重要決策反轉時，把舊邏輯標記為「歷史」而不是直接刪**
例如「按行銷樞紐」的金額一開始按 H 欄比例切（為了算獎金基數），但後來獎金獨立模組後，比例切分反而誤導使用者。改回完整案值，但保留「N 合作」案數標示。

**4. 先設計給人看，再優化給 AI 看**
中途有大改 — PM → 行銷、Sales → 業務、新增營運總監/編輯/總經理角色。系統內部 key 維持英文（PM/Sales/CEO），UI 全用中文。這個語言映射讓未來 AI 能繼續寫 code 不混亂。

## 七、踩到的坑，讓我更懂的事

### 坑一：多 PM 樞紐金額該不該按比例切？

**症狀**：原本想說多 PM 案的金額按 H 欄比例切，這樣每個行銷的「個人貢獻」就是獎金基數。但後來獎金結算改成「總經理主觀分配」，比例切分反而讓樞紐數字變得無意義。

這個問題讓我意識到：**UI 層的數字計算規則必須跟業務規則同步演化**。當業務規則改了，原本「合理」的算法可能就變成誤導。

> **帶走的原則**：每個自動計算的數字背後都有一條業務規則。規則變了，數字呈現方式也要重新檢視，不要堅持沿用「歷史的正確答案」。

---

### 坑二：Google Sheet 列號漂移 → 引入專案編號

**症狀**：原本獎金分配 tab 用「年度-列號」當識別（例如 `2026-15`）。但團隊在 Sheet 上手動編輯時，會插入空白列、刪除列，列號就漂移了，獎金記錄對到錯的專案。

**洞察**：使用者行為跟系統假設衝突時，**修系統而不是教使用者**。我們在 A 欄加「編號」（`P26-001` 自動產生），讓識別跟列號解耦。系統讀資料時自動 backfill 缺漏的編號，不需要手動維護。

> **帶走的原則**：把易變的東西（列號、列順序）跟穩定的東西（識別碼）分開。識別碼一旦發出去就不能重用、不能改，用「啟用 = 否」標記停用，而不是刪除。

---

### 坑三：選項散落各處 → 統一到「選項設定」tab

**症狀**：階段定義原本放在「目標設定」tab 的 H-J 欄（旁邊就是年度目標 A-F 欄）。要加 2027 目標時，視覺上很怪 —「下面就是空欄但右邊有另一張表卡著」。類別則完全沒中央定義，全靠手填，很容易打錯。

**洞察**：master list（人員、階段、類別等）需要一個專屬地方，不要塞進業務資料 tab。讓使用者改設定時不要怕改壞別的。

> **帶走的原則**：「設定」跟「資料」要分開放。即使現在數量小，未來只會更多。一個統一的選項設定 tab 能涵蓋階段、類別、未來新增的任何 enum，schema 簡單（類型/名稱/排序/啟用/描述）。

---

### 坑四：Modal 太擠 → 改成右側滑出 panel

**症狀**：專案明細用置中 Modal，內容太多時被切掉。試過縮字體、縮間距，還是不夠。

**洞察**：UI 的容器形式要符合內容性質。一個有 8-10 個欄位的詳細資訊頁面，本質上是「side detail view」，不是「dialog」。

> **帶走的原則**：Modal 適合「打斷主流程的決定」（例如確認刪除）。詳細資訊適合 side panel — 滑出、固定 header/footer、body 自然 scroll，捲軸合理出現。

---

### 坑五：權限分層（看到 ≠ 能編輯 ≠ 能登入）

**症狀**：所有員工都能登入儀表板沒問題，但獎金結算頁本來大家都看得到（雖然只有 admin 能編輯）。後來總經理要求只有他們能看到這個入口和頁面。

**洞察**：權限要分多層。`ALLOWED_EMAILS` 是「能登入」，`BONUS_ADMIN_EMAILS` 是「能看獎金結算」。不只是 API 拒絕，連 UI 入口也對非 admin 隱藏。

> **帶走的原則**：權限的「不顯示」比「顯示但拒絕」更好。讓非 admin 連入口都看不到，避免他們嘗試點擊後遇到拒絕訊息（讓他們更想知道裡面是什麼）。

## 八、AI 怎麼幫我做的

**分工方式**：
- **AI 主導**：所有程式碼、API 設計、UI 元件、debug 除錯
- **我主導**：業務規則、UI 取捨、優先順序、看到 demo 後決定要改什麼

**提問模式**：
- **症狀描述**：截圖 + 描述「這個地方怪怪的」（不寫程式術語，AI 自己診斷）
- **規則溝通**：用業務語言講需求，讓 AI 翻譯成資料模型 + UI
- **試錯確認**：AI 改完後我截圖回報，再決定接受 / 修正方向

**關鍵轉折**（讓任務有重大推進的時刻）：

1. **獎金規則重做** — 我以為是固定率，實際是池子 + 主觀分配。AI 主動建議重新討論資料模型，避免硬塞舊設計
2. **多 PM 比例的用途反轉** — AI 從程式角度問「比例現在不影響獎金，還要切金額嗎？」，幫我意識到該改回完整顯示
3. **編號 backfill 機制** — 我抱怨「列號會跑掉」，AI 提案「在 A 欄加編號 + 系統自動填」，比手動維護更穩

## 九、如果繼續往下

- **新增/刪除專案**：目前只能編輯既有專案，未來想從儀表板新增列、軟刪除（標記不顯示但保留歷史）
- **自動刷新**：目前手動點同步，可考慮 Sheets webhook 推播
- **通知與提醒**：待收款超過 N 天未收提醒、結案後 N 天未分配獎金提醒
- **獎金分配自動推薦**：累積足夠歷史資料後，AI 可以建議分配比例（總經理確認即可）

## Takeaway

**這個案例展示了什麼**：
- **資料源不變、介面層客製**：Google Sheet 留著當資料庫，介面層做所有複雜功能。團隊不用換工具
- **業務規則演化時，UI 邏輯跟著變**：原本的設計如果跟新規則衝突，**直接重做**而不是疊加 patch
- **權限分層**：能登入 ≠ 能看到所有頁面 ≠ 能編輯所有頁面，三層獨立白名單

**可移植性**：
- 適用於：任何「資料在 Google Sheet、想要更好介面」的情境
- 特別適合：業務規則複雜、會持續演化、團隊規模 5-30 人的公司
- 不適用：資料量極大（>10K 筆）、需要 row-level 權限、跨多個 app 共用資料

**如果你也要做，先問 AI 什麼**：
1. 「我有一個 Google Sheet，欄位是 [...]。我想做一個能讀取、篩選、編輯的儀表板。最簡單的技術組合是什麼？需要設定哪些金鑰？」
2. 「業務規則是 [...]。你會怎麼設計 Sheet schema？哪些欄位該放主表，哪些該獨立成 master list？」
